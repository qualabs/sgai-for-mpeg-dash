compound shape) that v6.1 introduces or modifies relative to the
baseline DASH 6th feature set defined in ISO/IEC 23009-1:2025. The
inventory mirrors v6's (C1..C30) so verdicts can be traced

quality (validate-spec sidecar's job), clarity, or alignment with
the context document.

## Marginal items detail

### M-1 (C16): Timescale units on linear-event integer attributes

**Status in v6.1**: explicit-documented, not resolved.

**Where in the spec**: v6.1 §2 NOTE at lines 162–179
(`<!-- refine: v6-dash-conformance-audit.md#M-1 -->`).

**v6.1's framing**: the NOTE states:

> The integer attributes `@maxDuration`,
> `@earliestResolutionTimeOffset`, `@returnOffset`, and
> `@clipDuration` on the inherited `<InsertPresentation>` /
> `<ReplacePresentation>` events (§5.1.1, §5.1.2) are declared in
> this specification as `xs:unsignedInt` expressed in the parent
> `<EventStream>@timescale` units. DASH 6th §5.16.3 / §5.16.4
> prose conventionally expresses the same attributes in seconds;
> the in-this-spec reading inherits the timescale-unit convention
> used throughout the annex examples (e.g. Annex A.2
> `maxDuration="20000"` with `timescale="1000"` means 20 s). A
> DASH-6th-baseline parser that reads the attributes as seconds
> would interpret an authored value of `20000` as 20000 seconds
> rather than 20 s — an interop risk worth flagging. Authoring
> tools targeting a mixed deployment SHOULD validate the intended
> interpretation against the receiving Player's edition and
> surface the timescale value alongside any externally exchanged
> attribute value to disambiguate.

**Audit reading**: the NOTE converts an undocumented assumption
into an explicit, auditable interop flag. That is the correct
authorial move at the v6.1 minor-refinement scale (the underlying
DASH 6th text remains unverified by NotebookLM). The Marginal
verdict stays because the canonical reading is still unresolved:
v6.1 commits to "timescale units"; DASH 6th prose
conventionally uses "seconds"; the two readings differ by 3
orders of magnitude at typical `timescale=1000`. A future audit
with working NotebookLM grounding (or direct PDF extraction)
must resolve which reading aligns with §5.16.3 / §5.16.4 prose
and either confirm v6.1's choice or schedule a major-build
correction.

`[fetch-failed]` on §5.16.3 / §5.16.4 attribute table
descriptions.

**Companion clarification — `@earliestResolutionTimeOffset` unit
context mismatch**: v6.1 also added a NOTE in §5.1.1 (refine:
detail-review flag-1) disambiguating that the same attribute name
on `<ImportedMPD>` (per §5.3.2.6 / §5.2.1.3 / §5.3.4) is in
seconds (inherited from §5.3.2.6 of the base specification),
whereas on slot-event children (§5.1.1, §5.1.2, §5.1.3.1,
§5.1.4.1) it is in `@timescale` units. This is internal hygiene
— it removes a same-name / different-units trap inside the spec
itself — and is unrelated to the open M-1 question (which is
about whether v6.1's `@timescale`-units reading matches DASH 6th
prose). The disambiguation NOTE is correct and well-placed.

## Conforming-with-comment items detail (selected, v6.1 changes only)

### CWC-1 (C3) → improved

v6.1 refined the §8.8 row for
`urn:svta:dash:profile:sgai-overlay-list:2026` to attribute the
gating mechanism to the parent event scheme (the §5.10
ignore-if-unknown rule) rather than to a non-existent "profile-
ignore" mechanism. The wording now reads:

> Gating is upstream at the main-MPD event scheme. DASH 6th
> carries no formal "profile-ignore" mechanism; `@profiles` is
> informative-hint level. The legacy Player never reaches this
> resolution document because the parent event scheme
> (`urn:svta:dash:event:sgai-overlay:2026` or
> `urn:svta:dash:event:sgai-pause-trigger:2026`) is unknown and
> gated by the §5.10 ignore-if-unknown rule.

The v6 critique is fully absorbed. Residual comment: the
end-to-end safety still depends on the upstream event scheme
being unrecognised; a Player that bypasses event-scheme gating
and reaches the document anyway will see an unknown `@profiles`
value, which DASH 6th treats as informative-hint only. The audit
verdict stays Conforming-with-comment for parallelism with v6 but
the comment is now baked into the spec rather than carried by
the audit.

### CWC-3 (C19) → improved

v6.1 added a NOTE in §5.3.4 documenting the namespace mixing on
`<svta:RenderableAsset>` children that include the core-namespace

baked into the spec.

### CWC-4 (C21) → improved

v6.1 added a NOTE in §5.5.2 documenting the validator-side
implication of placing the callback `<EventStream>` directly
inside `<svta:Candidate>`. The note states that validators
implementing this specification scan inside `<svta:Candidate>`
in addition to the DASH-canonical `<Period>` / `<MPD>` locations
when collecting callback events. The v6 critique is baked into
the spec.

### CWC-5 (C30) → resolved

v6.1 removed the `profiles="urn:mpeg:dash:profile:advanced-linear:2025"`

## Open questions surfaced (could not resolve from spec text alone)

The same five open questions from the v6 audit remain
unverifiable in this session because NotebookLM kept returning
off-topic answers:

1. **§5.16 attribute units (M-1)**: are `@maxDuration`,
   `@earliestResolutionTimeOffset`, `@returnOffset`, `@clipDuration`
   expressed in seconds or in `<EventStream>@timescale` units per
   DASH 6th §5.16.3 / §5.16.4 prose? v6.1's NOTE in §2 commits to
   timescale units and flags the interop risk explicitly, but the
   canonical reading remains unverified. `[fetch-failed]`.

2. **§8.14 ListMPD `MPD@type` value (was NC-1 fix path A)**: v6.1
   took fix path B (placeholder `<Period>`), so the question of
   whether §8.14 also defines a relaxed `@type="list"` value is
   no longer required to resolve NC-1. It remains relevant for a
   future major-build consolidation (whether to keep the
   placeholder-Period pattern or migrate to a `@type="list"`-style
   relaxation that would remove the placeholder anchor). The audit
   notes the open question but does not block.

3. **§5.10 `<EventStream>` placement (CWC-4)**: is placement of
   callback `<EventStream>` directly inside a foreign-namespace
   element (i.e. `<svta:Candidate>`) explicitly permitted by §5.10,
   or does it rely on the §5.2.1 foreign-namespace open-content
   rule alone? v6.1's §5.5.2 NOTE leans on §5.2.1; the §5.10-side
   question (does §5.10 explicitly admit non-canonical placements
   via its own clauses?) is unresolved. `[fetch-failed]`.

4. **Profile-ignore mechanism (CWC-1)**: does DASH 6th formally
   define behaviour for "Player encounters MPD with unknown
   `@profiles`"? v6.1's §8.8 refinement asserts that DASH 6th
   carries no formal mechanism — that assertion itself is the
   open question the audit cannot verify. `[fetch-failed]`.

5. **`PeriodType` foreign-content slot (new in v6.1 — C31)**:
   does the DASH 6th `PeriodType` complex type formally admit
   `<xs:any namespace="##other" processContents="lax"/>` directly
   inside the Period content model (alongside the
   `<AdaptationSet>`, `<EventStream>`, `<BaseURL>`, etc.
   children)? The v6.1 placeholder Period pattern depends on this
   slot. v6.1's §5.2.2.1 asserts that "`<Period>` permits
   foreign-namespace children per §5.2.1 of the base
   specification". If §5.2.1's foreign-namespace rule applies to
   every base element including `<Period>`, the assertion holds;
   if §5.2.1 admits foreign content only on a documented subset
   of base elements, the placeholder Period pattern needs
   re-evaluation. `[fetch-failed]`.
