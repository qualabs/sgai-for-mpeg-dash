## Gaps (8)

Things the spec defines but for which `context/` does not provide
enough information for a conformant implementation to ship
deterministically. Numbering follows v4.1 G-1..G-8; v4.1's G-9 is
rescinded as a false positive and v4.1's G-10 is closed. All
remaining gaps carry forward from v4.1 unchanged.

### G-1. No normative ADS obligation to author non-linear beacon `presentationTime` in the slot-window time base

- **Spec sections needing it**: §4.3 (A-1..A-7), §4.4.6, §5.5.2,
  §5.5.4, §7.5.
- **What context is missing**: `context/03-requirements.md` R13.2
  is a Player obligation ("the Player MUST execute the tracking
  schedule supplied by the ADS"), and §5.5.2 marks
  `<svta:RenderableAsset>`-child `<EventStream>` as the preferred
  non-linear carrier whose timing reference is "the Broadcaster-
  declared overlay window". The obligation falls structurally on
  whoever authors the beacons inside the resolution document —
  the ADS — yet §4.3 A-4 ("Carry ad-tracking beacons inside the
  resolution document … using `urn:mpeg:dash:event:callback:2015`")
  makes no statement about the time base the ADS MUST use.
  Result: a non-remapping ADS silently mis-fires quartile beacons
  and no actor is non-conformant. **Carries forward from v4.1
  unchanged.** v4.2's refinement-gaps row #3 still tags this as
  requiring an R13 split — outside minor scope.
- **What an implementer does today**: the ADS either guesses the
  intended time base, or the Player re-implements the remap
  unilaterally and risks contradicting §4.4.6's "fire at the
  positions declared" semantics.

### G-2. Click-through interaction is defined but the trigger event is not

- **Spec sections needing it**: §5.6.1 (`<svta:Click>` `@through`
  / `@trackingUrl`), §4.4 Player obligations, §7 Expected
  behaviour.
- **What context is missing**: no chapter binds "viewer
  interaction" to a concrete trigger surface per device class.
  `context/04-use-cases.md` enumerates D1..D5 rendering
  capabilities but no interaction surface;
  `context-analysis/iab-ad-templates.md` notes interactivity is
  *non-normative*. **Carries forward from v4.1 unchanged.**
  Refinement-gaps row #4.
- **What an implementer does today**: clicks on CTV remotes vs
  mobile gestures vs web pointer events are implementation-defined;
  cross-Player divergence on (a) surface, (b) `@trackingUrl`
  firing, (c) abort behaviour is outside the conformance surface.

### G-3. Pause-ad dismissal on viewer resume is informal

- **Spec sections needing it**: §5.1.4 attribute table for
  `<svta:PauseAdPresentation>`, §4.4 Player obligations, §7.4
  compositing rules, Annex E.
- **What context is missing**: UC-05 prose and Annex E.4
  ("dismissed on resume") describe the intended behaviour, but
  no normative section of the spec carries a "Player MUST dismiss
  the pause-ad when the viewer resumes primary playback" rule.
  §5.1.4 attribute `maxDuration` description ("applied from the
  moment the ad becomes visible until automatic dismissal") and
  `triggerMode` define no dismissal trigger besides the cap.
  **Carries forward from v4.1 unchanged.** Refinement-gaps
  row #5.
- **What an implementer does today**: dismisses on resume (the
  obvious reading) — but a stricter implementer reading just §5.1.4
  may hold the pause-ad until `maxDuration` expires regardless of
  resume.

### G-4. Pause inside an active `<svta:OverlayPresentation>` is unspecified

- **Spec sections needing it**: §4.4.7 single-form concurrent
  rendering, §4.4.6 tracking, §5.5.4 beacon timing, §7.4 / §7.5.
- **What context is missing**: `context/04-use-cases.md` keeps
  UC-03 and UC-05 mutually exclusive and never composes them. If
  the viewer pauses while an overlay form is visible, the
  overlay's slot-window clock (§5.5.4 — referenced to the
  Broadcaster-declared window) collides with the paused primary
  timeline: does the slot-window clock pause with primary, or
  continue against wall-clock? Quartile beacons fire on different
  schedules under the two readings. **Carries forward from v4.1
  unchanged.** Refinement-gaps row #6.
- **What an implementer does today**: ad-hoc. A Player that ties
  overlay timing to primary `presentationTime` will pause beacons;
  one that uses wall-clock keeps firing while the user is paused.

### G-5. Tracking-only ad (no `<MediaFile>`) lacks a normative carrier

- **Spec section needing it**: §8.2 (non-normative); §4.3 A-2
  ("populate every non-linear candidate with one or more
  renderable forms").
- **What context is missing**: `context/05-dash-linear-interfaces.md`
  flags this as "ADS-internal policy until a normative reference
  emerges from the IAB". §8.2 inherits that posture and recommends
  silent skip — but a tracking-only `<Ad>` is a real industry
  pattern (analytics-only impressions, server-side fallbacks).
  R6.2 / R13.4 assume an `<EventStream>` of the callback scheme
  attached to either a sub-MPD `<Period>` or an
  `<svta:RenderableAsset>` — both presuppose a renderable form.
  **Carries forward from v4.1 unchanged.** Refinement-gaps
  row #7.
- **What an implementer does today**: silently drops the
  impression. Production deployments fire these as bare HTTP GETs
  through vendor side channels.

### G-6. R8 per-construct justification audit table is not consolidated

- **Spec sections needing it**: §5.1.3, §5.1.4, §5.2.2, §5.3, §5.6
  (every new construct).
- **What context is missing**: `context/03-requirements.md` R8.1
  ("Every new construct introduced by the proposal MUST be
  accompanied by an inline justification stating why an existing
  MPEG-DASH construct could not be reused") and
  `context/07-backward-compat-checklist.md` request a
  per-construct audit. v4.2 still carries scattered design
  rationale (carrier-closure list in §4.2 B-4; "no new tracking
  scheme" in §5.5; "not a `ListMPD`" in §5.2.2; "DASH 6th defines
  no native carrier" in §5.6) but no single per-construct audit
  table. **Carries forward from v4.1 unchanged.** Refinement-
  gaps row #8.
- **What an implementer does today**: nothing fails at runtime;
  the gap is governance. A WG reviewer auditing R8 compliance has
  to reconstruct the justification by reading the gap analysis
  instead of finding it inline.

### G-7. SGAI `@earliestResolutionTimeOffset` keeps a colliding name with `<ImportedMPD>` despite context/06 mandating a rename

- **Spec sections needing it**: §5.1.3, §5.1.4 attribute tables.
- **What context is missing**: `context/06-naming-and-namespaces.md`
  explicitly mandates renaming `@earliestResolutionTimeOffset`
  when re-used on an SGAI element with a different unit. v4.2
  retains the v4.1 prose-patch approach (the §5.1.3 attribute
  description carries an inline disambiguation paragraph against
  `<ImportedMPD>@earliestResolutionTimeOffset`) but does NOT
  rename the attribute. The §5.1.2 NOTE inherited from v4.1
  reinforces that the unit on the SGAI side matches §5.16.5 of
  DASH 6th — which is internally consistent with the linear
  baseline but still leaves `context/06`'s "rename" mandate
  unmet. **Carries forward from v4.1 unchanged.** Refinement-
  gaps row #2.
- **What an implementer does today**: reads the inline
  disambiguation paragraph and gets the unit correctly; a reader
  who skims the §5.1.3 attribute table without the paragraph will
  mis-key it to seconds (the `<ImportedMPD>` reading).

### G-8. `<svta:RenderableAsset>`-child `<EventStream>` reinterprets baseline timing without a context-level authorisation

- **Spec sections needing it**: §5.5.3, plus the
  `context/08-dash-extension-rules.md` "extension-points" closure.
- **What context is missing**: `context/08-dash-extension-rules.md`
  enumerates the admissible carriers (DR-1..DR-6) but says nothing
  about the pattern "baseline DASH element placed under an
  SVTA-namespaced parent, timing reinterpreted to the parent's
  lifecycle". v4.2's §5.5.3 NOTE (inherited from v4.1) acknowledges
  the reinterpretation but cites only the §5.2.1 fallback model,
  not an explicit context authorisation. Under R1.3 ("MUST NOT
  alter or override the semantics of any pre-existing MPEG-DASH
  6th edition construct") this is borderline: the spec re-uses
  the `<EventStream>` carrier but redefines its `presentationTime`
  reference frame. **Carries forward from v4.1 unchanged.**
  Refinement-gaps row #15.
- **What an implementer does today**: parses the NOTE, applies
  the form-visible timing reference. The risk is conceptual /
  governance (R1.3), not runtime.

## Edge cases (7)

Boundary conditions not contemplated in `context/04-use-cases.md`
but that deployable Players will encounter. All seven carry
forward from v4.1 unchanged — the v4.2 refinement did not
address edge-case-class issues.

### EC-1. `allowedLayouts` populated entirely with inadmissible tokens

- **Trigger**: Broadcaster authors `<svta:OverlayPresentation
  allowedLayouts="linear pause-ad"/>` (per §5.1.3 layout-token
  table, both tokens are explicitly inadmissible inside
  `<svta:OverlayPresentation>`).
- **Why it matters**: §3.2 says the Player "MUST treat the form
  as if its layout were inadmissible and skip it (§4.4.5)" —
  per-form. The slot-level case where no form on any candidate
  could ever be admissible collapses to §8.1 E6 (no renderable
  form) and then to fall-through. The spec never identifies the
  malformed `allowedLayouts` as the proximate cause; the Player
  sees an ordinary E6. **Carries from v4.1 unchanged.**
  Refinement-gaps row #9.
- **Responsible actor / context silence**: Broadcaster authoring.
  Context silent on whether the Player should surface a
  diagnostic distinguishing "no candidate fit" from "no candidate
  could ever fit".

### EC-2. Overlay window exceeds linear window in a hybrid slot

- **Trigger**: `<ReplacePresentation>` declares
  `maxDuration="20000"` and the co-located
  `<svta:OverlayPresentation>` declares `maxDuration="30000"` —
  the linear ad ends 10 s before the overlay's declared window.
- **Why it matters**: §5.1.5 says "the shorter window governs",
  but the resulting composition becomes a single composition
  window rather than two independent slots — contradicting
  §5.1.5's "two independent events". Annex D walks only the
  overlay-shorter case. **Carries from v4.1 unchanged.**
  Refinement-gaps row #10.
- **Responsible actor / context silence**: Player. Context routes
  through generic §5.1.5 rule.

### EC-3. Every candidate semantically empty after E8 filtering routes through E6

- **Trigger**: A resolution document where every candidate's
  forms carry `@mediaType="video/webm"` (outside admissible set).
  After §8.1 E8 filtering, every candidate is empty; the Player
  walks each, finds no admissible form, and falls through under
  E6.
- **Why it matters**: §7.3 step 5 (1)-(2) processes per-candidate
  admissible form sets; an empty set after filtering triggers E6.
  If *every* candidate ends up empty after E8 filtering, the slot
  becomes a no-fill and §8.1 E3 / §6.2 (empty document) semantics
  should arguably apply retroactively. The spec routes this
  through E6 fall-through — operationally fine but operator
  diagnostics see "no renderable form" instead of "non-conformant
  ADS". **Carries from v4.1 unchanged.** Refinement-gaps
  row #11.
- **Responsible actor / context silence**: ADS conformance vs
  no-fill telemetry; the spec collapses them.

### EC-4. Pause inside an active overlay window

- **Trigger**: A `<svta:OverlayPresentation>` event window
  [120 s, 140 s] is rendering on D1; the viewer pauses at 125 s
  while a `<svta:PauseAdPresentation>` window [60 s, 660 s] is
  also active.
- **Why it matters**: §4.4.7 forbids simultaneous non-linear
  forms, but the spec does not say which form yields. Does the
  active overlay finish before the pause-ad request is issued?
  Is the overlay dismissed to allow the pause-ad? Does the
  pause-ad request fire at all while another non-linear form is
  rendering? Slot-window timing of the running overlay (§5.5.4,
  referenced to Broadcaster-declared window) collides with the
  paused primary timeline (see G-4). **Carries from v4.1
  unchanged.** Refinement-gaps row #6 (same root cause).
- **Responsible actor / context silence**: Player. Neither §4.4.7
  nor §7 addresses the resolution rule.

### EC-5. `maxDuration=0` on a slot

- **Trigger**: Broadcaster authors `<svta:OverlayPresentation
  maxDuration="0" .../>` (admissible per `xs:unsignedLong`).
- **Why it matters**: Every candidate's `@duration` would exceed
  the cap by definition → drop-before-play drops every
  candidate → fall through. The spec admits this as a degenerate
  but valid case without saying so explicitly. A Broadcaster
  could mis-author and the Player silently no-ops. **Carries
  from v4.1 unchanged.** Refinement-gaps row #12.
- **Responsible actor / context silence**: Broadcaster authoring;
  context silent.

### EC-6. ADS resolution returns the wrong document profile for the slot type

- **Trigger**: ADS responds to a `<ReplacePresentation>`
  resolution request with an Overlay Resolution Document, or to
  an `<svta:OverlayPresentation>` resolution request with a
  `ListMPD`.
- **Why it matters**: §4.3 A-1 ties the response shape to the
  slot type, so the mismatched response is non-conformant. §8.1
  E2's wording reads as if every parseable document maps to a
  valid resolution; the shape-mismatch case (well-formed
  document, wrong profile for the slot) is not surfaced. The
  cross-type mismatch lacks an explicit Player rule. **Carries
  from v4.1 unchanged.** Refinement-gaps row #13.
- **Responsible actor / context silence**: ADS conformance vs
  Player robustness. Context routes through generic E2.

### EC-7. Sub-MPD `<Period>@duration` vs `<svta:RenderableAsset @duration>` mismatch (video form)

- **Trigger**: A video form declares `@duration="PT20S"` but its
  child `<ImportedMPD>` resolves to an SPS sub-MPD whose
  `<Period>@duration="PT22S"`. Both attributes exist; both are
  Required.
- **Why it matters**: §5.3.1 `@duration` "drives the Player's
  drop-before-play arithmetic". §5.4 says `<Period>@duration`
  "declares the ad's intrinsic length". For cap arithmetic,
  drop-before-play uses the form-level value (20 s); for
  trim-during-play, the Player uses actual rendered length, which
  the sub-MPD authors as 22 s. The two attributes can disagree
  and the spec does not say which is canonical. **Carries from
  v4.1 unchanged.** Refinement-gaps row #14.
- **Responsible actor / context silence**: ADS authoring vs
  Player resolution. Context silent on the cross-document
  consistency.

## Ambiguities (2)

Places where the spec admits two readings because the context
language permits both. Both v4.1 ambiguities carry forward; v4.2
introduced none.

### A-1. ADS responsibility to remap non-linear `presentationTime` is implied but not codified

- **Context passage**: `context/03-requirements.md` R13.2 — "the
  Player MUST execute the tracking schedule supplied by the ADS"
  (i.e. the Player is bound to fire the carrier's
  `<Event @presentationTime>` values verbatim).
- **Reading 1**: Player remaps inbound beacons against the slot
  window — re-computes quartiles, ignoring the ADS-emitted
  `presentationTime` values.
- **Reading 2**: ADS authors `presentationTime` in slot-window
  time so the Player fires the beacons verbatim. This is what
  §5.5.2 ("preferred placement … timing reference: overlay
  window") and §5.5.4 ("quartile beacons at 25 %, 50 %, 75 %,
  100 % of the Broadcaster-declared overlay window") assume — but
  §4.3 A-4 stops at "use the callback scheme" and never demands
  that the time base be slot-window.
- **Spec assumption**: Reading 2 — but without an ADS-side
  normative obligation. See G-1.
- **Tighter context sentence**: split R13.2 into an ADS MUST
  (emit `presentationTime` against the Broadcaster-declared slot
  window for non-linear ads) and a Player MUST (fire verbatim).
  **Carries from v4.1 unresolved.**

### A-2. `<svta:RenderableAsset>` reinterprets baseline `<EventStream>` semantics without context authorisation

- **Context passage**: §5.5.3 — "the `<Event>@presentationTime`
  is interpreted relative to the moment the form becomes visible
  (i.e. the impression instant)." `urn:mpeg:dash:event:callback:2015`
  is defined by DASH 6th §5.10 with `presentationTime` relative
  to the carrying `<Period>` start.
- **Reading 1**: The spec reuses the callback scheme's syntax but
  redefines its `presentationTime` reference frame when the
  `<EventStream>` is a child of `<svta:RenderableAsset>` instead
  of `<Period>`. Under R1.3 ("MUST NOT alter or override the
  semantics of any pre-existing MPEG-DASH 6th edition construct")
  this is a borderline violation — the carrier is reused but its
  timing semantics are not.
- **Reading 2**: The reference frame shift is intrinsic to placing
  `<EventStream>` outside a `<Period>` parent; there is no
  baseline rule about non-`<Period>`-parent `<EventStream>` so
  the spec is free to define one. The §5.5.3 NOTE inherited from
  v4.1 formalises Reading 2 inside the spec.
- **Spec assumption**: Reading 2. The §5.5.3 NOTE clarifies
  intent — but the foreign-namespace open content rule (§5.2.1)
  admits *new SVTA elements* under existing parents; it does not
  authorise placing a *baseline DASH element* (`<EventStream>`)
  under a *new SVTA-namespaced parent* with redefined timing.
- **Tighter context sentence**: `context/08-dash-extension-rules.md`
  or `context/06-naming-and-namespaces.md` should explicitly
  admit the pattern "baseline DASH element placed under an
  SVTA-namespaced parent, timing reinterpreted to the parent's
  lifecycle". **Carries from v4.1 unresolved.** See G-8.

## Recommended context refinements

Top 5 by leverage, ordered by the number of downstream items each
unblocks. **All five carry from v4.1 unresolved** — the v4.2
refinement deliberately stayed inside spec-only scope and did not
attempt context-level changes.

### 1. Split R13.2 into ADS MUST and Player MUST

`context/03-requirements.md` R13.2 binds the Player to execute
the ADS-supplied schedule. The spec assumes the ADS authors
`<Event @presentationTime>` against the Broadcaster-declared slot
window for non-linear ads, but no ADS obligation states this.
Add an ADS-side requirement (e.g. R13.2a — "for non-linear slots,
the ADS MUST emit callback `<Event @presentationTime>` values
measured against the Broadcaster-declared slot window, not the
ad's intrinsic duration") and keep R13.2 as the Player-side
complement. Closes G-1 and A-1; unblocks R13 → full and tightens
§4.3 with a new A-N row. **High-leverage**: single context edit,
multiple downstream effects.

### 2. Define the Player obligation to dismiss `<svta:PauseAdPresentation>` on viewer resume

`context/04-use-cases.md` UC-05 prose describes the dismissal but
no requirement captures it. Add a Player MUST rule to R14 or a
new R that fires on the pause-state transition ("when the viewer
resumes primary playback, the Player MUST dismiss any active
pause-ad form and MUST stop firing further beacons"). Closes G-3
and clarifies the pause-ad lifecycle independent of the cap.
Touches a new §4.4.x sub-section and Annex E.

### 3. Decide rename vs admit-NOTE for `@earliestResolutionTimeOffset`

`context/06-naming-and-namespaces.md` mandates renaming on
collision with `<ImportedMPD>`; v4.2 keeps the prose-patch
approach. Either rename the SGAI attribute (e.g.
`@earliestResolutionTimeOffsetTicks`) to make the unit audible at
the XML call site, or amend `context/06` to admit "rename OR
explicit inline NOTE" as a permitted audibility lever. Closes
G-7. Touches §5.1.3, §5.1.4, every annex with overlay or
pause-trigger events.

### 4. Compose UC-03 and UC-05 (pause during active overlay)

`context/04-use-cases.md` keeps UC-03 and UC-05 mutually
exclusive even though both ride on the primary timeline. Extend
UC-03 with a "viewer pauses during the overlay window"
sub-scenario and pick a rule for the slot-window clock (pause
with primary, or continue against wall-clock). Closes G-4 / EC-4
and forecloses divergent Player implementations on a real
production trigger (any CTV remote pause during a non-linear
ad).

### 5. Admit the "baseline element under SVTA parent with reinterpreted timing" pattern in `context/08-dash-extension-rules.md`

The current DR-1..DR-6 closure of `context/08` does not enumerate
the §5.5.3 / §5.3.4 pattern (baseline DASH element placed inside
a foreign-namespace parent, timing reference reinterpreted). The
v4.1/v4.2 NOTES describe the pattern operationally; the
context-level authorisation is still missing. Add a DR-N entry
stating "a baseline DASH element MAY appear inside an
SVTA-namespaced parent when the parent is foreign-namespace open
content, with a timing reference reinterpreted to the parent's
lifecycle; the §5.2.1 fallback model preserves legacy-Player
ignore semantics". Closes G-8 / A-2.
