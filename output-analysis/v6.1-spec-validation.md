## Scope

v6.1 is the first minor refinement of v6. Per the build metadata
header, the iteration is "delta-only against v6-sgai-spec.md,
applying issues from v6-spec-validation.md, v6-detail-review.md, and
v6-dash-conformance-audit.md without changing requirements or
architecture; every edit annotated inline with `<!-- refine:
<issue-id> -->`". The spec itself enumerates 13 "Refinement gaps"
(items 1–13 of the trailing table) that could not be resolved at
minor-refinement scale because they require a `context/` change.

This validation walks v6.1 end-to-end against `../context/` and
`../context-analysis/` and:

1. Confirms which v6-findings the inline refinements actually closed.
2. Carries forward the v6 findings that the spec itself flags as
   unresolved (the 13 refinement gaps in the trailing table).
3. Surfaces new issues introduced by the v6 → v6.1 deltas.
4. Re-walks R coverage against the refined sections.

NotebookLM is not used in this spec-only validation pass per the
prompt; the per-construct DASH 6th conformance check is left to the
`audit-dash-conformance` sidecar.

## Gaps (5)

The first four gaps below are carried forward from v6 (G-1..G-4 in
`v6-spec-validation.md`); each is also enumerated as a refinement
gap in v6.1's trailing table (items 1–4). G-5 is new — a defect
introduced by the v6 → v6.1 refinement itself.

### G-1. Click-tracking beacon lifecycle after slot trim remains unspecified

**Spec section needing it**: §4.6.10, §5.6.1, §8.1 (E12), §8.3.

**Status against v6**: unchanged. v6.1 did not touch §5.6.1 or
§8.3; R13.3 still binds only schedule-driven beacons, leaving the
event-triggered click beacon under-specified for the
candidate-trimmed case.

**What an implementer would do today**: pick one of *fire
opportunistically* (consistent with E12 fire-and-forget) or
*suppress after trim* (consistent with R13.3 in spirit), producing
divergent click-tracking ledgers across deployments.

**Cross-ref**: v6.1 refinement gap #1.

### G-2. Default uncovered-region behaviour for `squeezeback-double-box` (no `-with-background`)

**Spec section needing it**: §3.2 (layout enum row),
§5.3.7 (background fill rules).

**Status against v6**: unchanged. §3.2's row for
`squeezeback-double-box` still says "filled with the Player's
default fallback"; §5.3.7's precedence ladder still ends with
"typically a uniform fill". `context/03-requirements.md` R26
treats side-by-side / double-box as **one** IAB layout with a
**MAY** background fill — the two-token split (`squeezeback-double-box`
vs `squeezeback-double-box-with-background`) is a spec invention
that R26 does not back. The implementer is left to ship a black
band, an opt-in uniform colour, or quietly conflate the two
tokens.

**Cross-ref**: v6.1 refinement gap #2.

### G-3. `<svta:RenderableAsset>` exposes layout tokens with no slot construct

**Spec section needing it**: §3.2 (vocabulary), §5.1 (event
constructs), §5.3.3 (enum), §7.5.

**Status against v6**: unchanged. §3.2 and §5.3.3 still enumerate
`menu-ad`, `screen-saver-ad`, `companion-ad` as admissible
`@layout` values, with slot-family annotations (`(CTV UI)`,
`(device-initiated)`, `(out-of-player)`) that map to no
`<EventStream>@schemeIdUri` introduced in §2.1. A Publisher has no
construct to declare a slot of any of these kinds in this edition;
an APS emitting a candidate with one of these `@layout` values has
no parent event to anchor against. The cleanest read of
`context-analysis/iab-ad-templates.md` is that the tokens are
**reserved for a future edition**; that decision is still not
stated in `context/`.

**Cross-ref**: v6.1 refinement gap #3.

### G-4. `<ImportedMPD>` placement inside a foreign-namespace parent — SPS-binding semantics still unstated

**Spec section needing it**: §5.3.4 (carrier for video forms),
DR-1 in `../context/08-dash-extension-rules.md`.

**Status against v6**: partially mitigated by a new NOTE (refine
anchor `v6-dash-conformance-audit.md#CWC-3` at §5.3.4) that
documents the **schema-author** implication — that a future SVTA
schema for `<svta:RenderableAsset>` MUST declare the core-namespace
`<ImportedMPD>` as an admissible child via `<xs:any
namespace="urn:mpeg:dash:schema:mpd:2011" processContents="lax"/>`
or explicit `<xs:import>`. The NOTE does **not** resolve the
upstream question the v6 validation raised: when `<ImportedMPD>`
is reached via a foreign-namespace parent (not the canonical
baseline `<Period>`), does the DR-1 SPS-binding chain (§5.3.2.6 →
§8.15 → §7.3 → RFC 4337) still apply? The CWC-3 NOTE is silent on
that semantics question. The implementer is left to assume the
binding travels with `<ImportedMPD>` itself; nothing in `context/`
confirms it.

**Cross-ref**: v6.1 refinement gap #4.

### G-5. Inline lowercase-codec NOTE in §5.4 is self-contradictory (new in v6.1)

**Spec section needing it**: §5.4 (sub-MPD codecs NOTE introduced
by `<!-- refine: v6-detail-review.md#flag-3 -->`).

**Issue (new in v6.1)**: the refinement added the following
sentence at §5.4 to address `v6-detail-review.md` flag-3 (mixed
case across examples):

> "Codec strings ... use lowercase hexadecimal in the codec
> identifier per IETF RFC 6381 §3 (`avc1.4d401f`, not
> `avc1.4d401f`)."

Both string literals between the parentheses are **identical**
(`avc1.4d401f`). The intended contrast is presumably
`avc1.4d401f` (lowercase, correct) vs `avc1.4D401F` (uppercase,
incorrect), but the typo collapses the example. A reader sees the
rule asserted without a worked contrast and cannot infer what the
disallowed form looks like.

**What an implementer would do today**: read the rule literally
and ship lowercase strings (the surrounding prose is unambiguous);
the typo is a documentation defect, not a correctness defect.

**Suggested fix**: replace one of the literals with the
uppercase form so the contrast lands — e.g. "`avc1.4d401f`, not
`avc1.4D401F`". This is mechanically applicable in a v6.2 minor
refinement.

## Edge cases (4)

All four edge cases are carried forward from v6 (EC-1..EC-4); v6.1
did not address any of them at the spec level. They appear in the
v6.1 trailing refinement-gaps table as items 5–8.

### EC-1. Hybrid slot where the overlay window is longer than the linear window

**Trigger**: Publisher authors a hybrid (UC-04) with co-located
events where the linear `@maxDuration` is shorter than the overlay
`@maxDuration`.

**Status against v6**: unchanged. §5.1.5 still states the two
events are "independent at authoring time and at runtime"; §7.5.4
walks the D1..D5 outcomes during the linear portion but is silent
on what happens after the linear ad ends and the overlay window
still has time on the clock. Independent-event semantics would
imply the overlay continues against the resumed primary content
until its own window expires, but no chapter says so.

**Responsible actor**: Publisher declares; Player enforces.
`context/04-use-cases.md` UC-04's "Notes / open questions" lists
the unresolved cross-portion-linkage question but not this
boundary.

**Cross-ref**: v6.1 refinement gap #5.

### EC-2. Pause-trigger window overlapping a linear ad break

**Trigger**: Publisher declares a `<svta:PauseAdPresentation>`
window whose presentation time straddles a co-scheduled
`<ReplacePresentation>` or `<InsertPresentation>` linear break.

**Status against v6**: unchanged. §4.6.7 (R16) and §4.6.8 (R17)
cover pause-ad-vs-overlay (R17 is cross-family); §4.6.11 (R20.1)
covers same-family overlap. A **linear break + pause-trigger**
overlap is cross-family but not pause-ad-vs-overlay, so neither
R17 nor R20 binds. §7.5.5 assumes the viewer is in primary content
when the pause occurs.

**Responsible actor**: Publisher declares; Player resolves.

**Cross-ref**: v6.1 refinement gap #6.

### EC-3. Empty resolution document on the first same-family overlapping window

**Trigger**: Two overlapping `<svta:OverlayPresentation>` windows;
the APS for the first responds with HTTP 200 and an empty
`<svta:OverlayList>` (§8.1 E2). R20.1 says fall-through to the
second window is admissible only when the first is **inaccessible**;
HTTP 200 with an empty body is accessible.

**Status against v6**: unchanged. The combination of R20.1
("cannot access") with §8.1 E2 ("empty resolution document is a
clean fall-through to primary") forces the Player to honour the
first window's empty document and leave the second window
unresolved. The viewer sees no ad. The conjunction is intentional
under a strict reading of R20.1 but produces a counter-intuitive
outcome.

**Responsible actor**: Player. The Publisher cannot author
around it.

**Cross-ref**: v6.1 refinement gap #7.

### EC-4. Background fill image fails to load mid-render

**Trigger**: The Player commits to a candidate's
`squeezeback-double-box-with-background` option; the
advertiser-supplied `<svta:BackgroundFill>@assetUrl` or the
Publisher's `@backgroundFallback` URL fetch fails after the
candidate has been admitted.

**Status against v6**: unchanged. §5.3.7's precedence ladder
covers source selection at commit time but not graceful
degradation after a third-element fetch failure mid-render.
§8.1 E11 ("cease rendering the candidate; advance to the next")
binds to ad-asset failures, not to background-image failures. The
Player is left to choose between (a) treating the failure as E11
and dropping the candidate, (b) degrading down the precedence
ladder to the next source, or (c) compositing the Player default
fallback (§3.2 row's "default fallback") with no candidate
discard.

**Responsible actor**: Player.

**Cross-ref**: v6.1 refinement gap #8.

## Ambiguities (6)

A-1..A-5 are carried forward from v6 (none were resolved at minor
scale — they all require `context/` decisions). They appear in the
v6.1 trailing refinement-gaps table as items 9–13. A-6 is **new**:
it was introduced by the v6 → v6.1 refinement of `@value` semantics.

### A-1. Spec-side ad-type identifiers diverge from the IAB mirror

**Spec passage**: §3.2 (Spec-side identifier column).

**Status against v6**: unchanged. The four diverging tokens
(`pause-ad`, `menu-ad`, `screen-saver-ad`, `companion-ad`) are
still spec-side names with a `-ad` suffix that does not appear in
`context-analysis/iab-ad-templates.md` (which proposes `pause`,
`menu`, `screensaver`, `companion`). R12.2 mandates a 1:1
mapping to IAB-defined ad-type values; the per-row mapping that
would let a reader audit "1:1 admits a consistent suffix
transformation" is still not shown in either place.

**Reading assumed by v6.1**: same as v6 — interpretation (i)
(the `-ad` suffix is a spec-side disambiguation move, the
underlying IAB mapping is 1:1).

**Cross-ref**: v6.1 refinement gap #9.

### A-2. `@maxConcurrency` admissible value is fixed at `1` — DP-1.1 tension

**Spec passage**: §5.1.3.1 (`<svta:OverlayPresentation>`
attribute table); same critique applies to
`<svta:PauseAdPresentation>@allowedLayouts` (§5.1.4.1) whose
admissible token is fixed at `pause-ad`.

**Status against v6**: unchanged. DP-1.1
(`context/03-requirements.md`) still forbids constructs whose
only admissible value matches the construct's default or is fixed
by another rule. `@maxConcurrency` with sole admissible value `1`
is precisely that case; the spec defends it inline as an
"authoring-time-declarative" choice, but `context/` does not state
the exemption.

**Cross-ref**: v6.1 refinement gap #10.

### A-3. Annex C / Annex I use `linear` inside `<svta:OverlayPresentation>@allowedLayouts`

**Spec passage**: Annex C.2 line 2243, Annex I.2 line 2748;
§5.3.3 (layout enum, "Slot families" column).

**Status against v6**: unchanged. The Annex C / I examples still
list `linear` as the last token in `@allowedLayouts` on an overlay
event; §5.3.3 still pins `linear` to "linear insert, linear
replace" in the "Slot families" column. The per-class walks in
§C.5 / §I.4 rely on `linear` being a valid takeover-fallback,
implicitly making the "Slot families" column advisory — but the
spec never says so.

**Cross-ref**: v6.1 refinement gap #11.

### A-4. §4.6.11 enumerates "late arrival" as a fallback trigger; R20.1 says "cannot access"

**Spec passage**: §4.6.11 (overlapping same-family windows);
`context/03-requirements.md` R20.1.

**Status against v6**: unchanged. §4.6.11's "transport failure,
malformed response, late arrival" enumeration still broadens
R20.1's "cannot access" wording. The spec carries the broader
list; `context/` carries the narrower one. The reader has to
infer that R20.1's "cannot access" subsumes "arrived after the
slot has elapsed", which is defensible but not stated.

**Cross-ref**: v6.1 refinement gap #12.

### A-5. Tracking carrier MAY live in two places for video forms

**Spec passage**: §5.5.2 ("MAY live inside the candidate's video
sub-MPD or directly inside the `<svta:Candidate>`"); §5.4.2
("the union of beacons is fired by the Player; duplicate beacons
... are de-duplicated").

**Status against v6**: unchanged. R13.1 still says "the APS
exercises no discretion over the schedule — it neither adds,
removes, nor reorders beacons". v6.1's §5.5.2 / §5.4.2
union-with-dedup model effectively admits **two-location
authoring** with Player-side de-duplication; R13.1 does not
contemplate authoring-location flexibility, and the de-duplication
key (`@id` and the URL+`@presentationTime` pair) is asserted in
§5.4.2 but not grounded in `context/`.

**Cross-ref**: v6.1 refinement gap #13.

### A-6. `<EventStream>@value` for callback scheme is now informational with any-string admissibility (new in v6.1)

**Spec passage**: §5.5.1 attribute table row for `@value`
(introduced by `<!-- refine: v6-dash-conformance-audit.md#M-2 -->`).

**Two readings**:

(i) The refinement closes M-2 by saying explicitly that the
Player does not key de-duplication on `@value` and any string is
admissible. The convention `value="1"` is documented as
*recommended for readability*, not normative.

(ii) The `Required` column for `@value` still reads "yes"
(inherited from §5.10), and yet the row now states that any
string is admissible *and* that the Player ignores the value
altogether at the level of de-duplication semantics. An attribute
that is required for syntax and ignored for semantics straddles
the DP-1.1 "no future-flexibility placeholders" rule: a mandatory
attribute whose only effect is readability looks like the kind of
construct DP-1.1 forbids.

**Reading assumed by v6.1**: (i), with the inline note as the
defence.

**Tighter context sentence**: the spec or `context/` should
either (a) state explicitly that the `Required` column for
`@value` is inherited verbatim from DASH 6th §5.10 (and that this
SGAI edition cannot relax it without amending the base scheme),
or (b) revise R23 / R6 to admit "attributes inherited from the
baseline DASH scheme MAY be mandatory at the syntax level and
informational at the SGAI semantics level". Either makes the
DP-1.1 boundary explicit; today the reader has to infer it.

   click-tracking beacons fire on viewer interaction after the

   names the idiom, or by pulling `linear` out of the Annex C / I
   examples. Without this, every Annex-C-style example reads as a
   contradiction with §5.3.3's "Slot families" column.

   `squeezeback-double-box-with-background` token split (G-2)** in
