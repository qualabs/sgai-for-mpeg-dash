# DASH conformance audit — v4.2 (2026-05-15)

**Audit target**: `../output/v4.2-sgai-spec.md`.
**Reference**: MPEG-DASH 6th edition (ISO/IEC 23009-1:2025, FDIS),
  consulted via NotebookLM notebook "Streaming Protocols — DASH,
  HLS, C2PA, DRM" (the locally registered notebook bound to the
  `NOTEBOOK_NAME=streaming specs` setting in `.env.agent`).
**Method**: inventory of new / modified constructs introduced by
  the spec, NotebookLM queries grounded against the DASH 6th
  source, per-construct verdict
  (Conforming `V` / Marginal `M` / Non-conforming `N`).

## Scope

v4.2 is a **minor refinement** of v4.1. `context/` is unchanged;
the spec was regenerated via `prompts/refine-spec.prompt` to
apply remedies for items the v4.1 conformance audit, v4.1
detail review, and v4.1 validation sidecar flagged as actionable
inside the minor envelope. The autofix surface in v4.2 (every
refine delta is annotated with an inline `<!-- refine: <issue-id> -->`
HTML comment) covers:

- **M-5** (v4.1 audit) — `<UrlParamInfo>` vs `<RequestParam>`
  element-name authority: §2 normative references and §5.7 now
  carry inline NOTEs documenting that DASH 6th carries the same
  element under two prose names (§I.3.1 prose `<UrlParamInfo>` vs
  clause 5.3 semantic tables 3 / 4 / 16 / 44 `<RequestParam>`).
  The spec follows the §I.3.1 prose name throughout and flags the
  inconsistency as an erratum candidate against the MPEG-DASH
  editor. The mitigation matches the M-1 pattern (inline NOTE +
  prose-fidelity choice + erratum flag) the v4.1 audit recommended.
- **G-10** (v4.1 validation) — §5.1.2 now carries an inline NOTE
  pinning the `@returnOffset` (`xs:unsignedLong`, no default, in
  `<EventStream>@timescale` units, measured from the slot's
  `presentationTime`) and `@startWithOffset` (`xs:boolean`, default
  `false`, controls whether the alternative starts at the offset
  or from its origin) semantics. The Annex B.2 / D.2 / F.2
  examples already emit `returnOffset="0" startWithOffset="false"`
  (carried over from the v4.1 N-2 autofix); the new NOTE removes
  the reader's need to follow the "defined verbatim in §5.16.4"
  pointer.
- **G-7** (v4 validation) — §5.1.2 adds a second inline NOTE
  clarifying that `@earliestResolutionTimeOffset` and `@maxDuration`
  on `<InsertPresentation>` / `<ReplacePresentation>` are in
  `<EventStream>@timescale` units per §5.16.5, matching the
  SGAI-side convention used on `<svta:OverlayPresentation>` and
  `<svta:PauseAdPresentation>`. This audit grounds the unit claim
  against DASH 6th — see Q1 below — and confirms it.
- **detail-review flag-1** (v4.1) — §5.1.4
  `<svta:PauseAdPresentation>@allowedLayouts` row in the attribute
  table now reads "Required=Yes, Default=`—`" (the v4.1 row had
  the contradictory pair "Required=Yes, Default=`pause-ad`"). No
  DASH-conformance impact (the attribute is foreign-namespace).
- **detail-review flag-2** (v4.1) — §5.3.3 trims the admissible
  `@layout` tokens for the pause-trigger scheme to `pause-ad`,
  `overlay`, `overlay-corner`, `overlay-lower-third` (the v4.1
  row inadvertently admitted `squeezeback-*` tokens, which
  contradict the paused-frame use case). No DASH-conformance
  impact (the enum is SGAI-side).
- **detail-review flag-3** (v4.1) — §5.6.3 wording cleanup (drops
  a dangling cross-reference to §2.1). No DASH-conformance
  impact.

v4.2 carries forward, by explicit declaration in the spec body's
"Refinement gaps" section, all v4.1 items the minor envelope could
not resolve — most notably **N-1** (Overlay Resolution Document
with zero `<Period>` elements) and **M-2**
(`@earliestResolutionTimeOffset` rebound unit on
`<svta:OverlayPresentation>` / `<svta:PauseAdPresentation>`). The
audit re-asserts the unchanged status of these items.

This audit is conformance-against-DASH-6th only. Design-level
audit lives in the v4.2 spec validation sidecar
(`v4.2-spec-validation.md`); detail review lives in
`v4.2-detail-review.md`.

## NotebookLM findings (highlights)

One fresh NotebookLM query against the DASH 6th notebook grounds
the substantive v4.2-new verdicts.

**Q1 — §5.16.5 unit declaration on `@earliestResolutionTimeOffset`
and `@maxDuration` (verifies the v4.2 §5.1.2 G-7 NOTE).**
NotebookLM quoted the §5.16.5 "Common Alternative Media
Presentation Properties" semantic table verbatim:

- `@earliestResolutionTimeOffset` — *"specifies the time interval
  (in units of timescale) prior to the Event@presentationTime
  during which the MPD described in the @uri attribute may be
  requested. This interval is expressed in units of
  EventStream@timescale"*.
- `@maxDuration` — *"specifies maximum duration of the
  Alternative Presentation, expressed in units of
  EventStream@timescale."*.

The v4.2 §5.1.2 NOTE (lines 699–707) matches this verbatim and
the unit-comparison claim with `<svta:OverlayPresentation>` /
`<svta:PauseAdPresentation>` (which use the same time base) is
confirmed.

NotebookLM also surfaced a DASH-6th-internal oddity: the same
table cell notes *"The default value is 60s (in units of
timescale)."* — the literal "60s" reads as "60 seconds" but the
parenthetical says "in units of timescale". This is an inherited
DASH 6th prose discrepancy independent of SGAI, registered here
for visibility but not actioned by the audit. (The v4.2 spec
does not propagate this confusion: the SGAI worked examples set
`earliestResolutionTimeOffset="10000"` against
`timescale="1000"`, which is unambiguous regardless of the
"60s" prose anchor.)

**Q2 (same call) — `<xs:element>` declarations for `UrlParamInfo`
and `RequestParam` in the DASH 6th schema (extends v4.1 Q3 for
C-28).** NotebookLM confirmed: *"Neither
`<xs:element name=\"UrlParamInfo\">` nor
`<xs:element name=\"RequestParam\">` is actually declared in the
normative XML schema fragments visible in the text."* The
fragments quoted for `MPDType` and `EventStreamType` run from
their named children (`Metrics`, `EssentialProperty`,
`SupplementalProperty`, `UTCTiming`, `LeapSecondInformation`,
`ContentSteering` for `MPDType`; `Event` for `EventStreamType`)
directly to the `<xs:any namespace="##other" processContents="lax">`
wildcard, with no `RequestParam` / `UrlParamInfo` declaration in
the printed schema at all.

This is sharper than v4.1 Q3, which said only that "the
`<xs:element>` declaration is omitted from the quoted schema
snippets". Q2 establishes that the omission spans **both** names
across every relevant `<xs:sequence>` block visible in the text.
A strict schema validator running off the printed schema would
reject **either** name as a natively declared element in the
`urn:mpeg:dash:schema:mpd:2011` namespace. The choice between
`<UrlParamInfo>` and `<RequestParam>` is therefore not a
schema-validator question (neither passes the strict
validator on the printed schema alone); it is purely a
prose-fidelity question, which v4.2's §2 NOTE handles
adequately.

The on-disk DASH-MPD.xsd published alongside ISO/IEC 23009-1:2025
may declare the element under one name or the other (or both,
or use a foreign-namespace shim) — the spec text does not show
this, so the audit treats the printed schema as authoritative
for what a reader can verify from the spec.

## Non-conforming items detail

### N-1: Overlay Resolution Document has zero `<Period>` elements (C-20)

**Status: carried forward unchanged from v4.1 audit.** The v4.2
spec acknowledges the gap explicitly in the spec body's
"Refinement gaps" section (row 1) and defers the fix to the next
major build.

The §5.2.2 boilerplate (lines 914–931), Annex C.3, and Annex E.3
all emit a `<MPD>` whose only child is `<svta:OverlayList>` —
zero `<Period>` elements. Per the §5.3.1.2 / §5.3.1.3 schema,
`<Period>` is `maxOccurs="unbounded"` with implicit `minOccurs="1"`.
A strict DASH-conformant XML schema validator would reject the
document.

**Why minor refinement could not lift this**: every proposed
remedy is an architectural change to the resolution-document
shape — wrap `<svta:OverlayList>` in a `<Period>`; place it
inside an AdaptationSet-bearing `<Period>`; or abandon the
`<MPD>` envelope altogether — and each rewrites §5.2.2, every
non-linear annex, and the
`urn:svta:dash:profile:sgai-overlay-list:2026` profile
constraint. The Refinement-gaps row 1 framing is unchanged from
v4.1: the choice between the three remedies is a context-level
architectural decision deferred to the next major build.

**Suggested fix for the next major build**: the cleanest of the
three is the "Period-wrapper" variant — declare a single empty
`<Period id="overlay-resolution" duration="P0S">` and place
`<svta:OverlayList>` as its child under §5.2.1 foreign-namespace
open content. This re-uses the established §5.2.1 extension path
already exercised everywhere else in the spec, makes the
document schema-conformant against `<Period minOccurs="1">`, and
costs only one syntactic level of nesting. (Variant trade-offs
are out of scope for this audit; see Refinement-gaps row 1.)

## Marginal items detail

### M-1: `@url` vs `@uri` on `<InsertPresentation>` / `<ReplacePresentation>` (C-14, C-15)

**Status: carried forward unchanged from v4.1 audit.** v4.2
§5.1.1 (lines 652–663) and §5.1.2 (lines 679–684) preserve the
v4.1 inline NOTEs spelling out the DASH-6th editorial split:
schema declares `<xs:attribute name="uri" type="xs:anyURI" use="required"/>`
on `AlternativeMPDEventType`; Table 62 prose names the same
attribute `@url`. The NOTEs flag the inconsistency back to the
MPEG-DASH editor as an erratum candidate. A reader using a
strict schema validator is forewarned.

The spec continues to author `url="…"` in §5.1.1, §5.1.2, and
Annex A.2, so the schema validator would still reject the
example XML. The trade-off remains as in v4.1: prose-fidelity to
the linear SGAI baseline vs. schema fidelity. The v4.1 verdict M
is preserved with the documentation now stable.

### M-2: `@earliestResolutionTimeOffset` rebound unit on SVTA elements (C-13)

**Status: carried forward unchanged from v4.1 audit.** v4.2
preserves the §5.1.3 attribute-table inline disambiguation
paragraph (around line 727) and the SGAI-side convention table.
v4.2's new §5.1.2 NOTE (G-7 close, lines 699–707) **reinforces**
this position by stating that the linear-baseline
`@earliestResolutionTimeOffset` is itself in `<EventStream>@timescale`
units per §5.16.5 (grounded by Q1 of this audit), so the SGAI
unit is consistent with the linear baseline.

The remaining issue is the cross-construct collision with
`<ImportedMPD>@earliestResolutionTimeOffset`, which keeps the
seconds unit. `context/06-naming-and-namespaces.md` still
mandates a rename when an attribute is reused with different
semantics; v4.2 leaves the rename unmet.

**Why minor refinement could not lift this**: the
context-level rule in `context/06-naming-and-namespaces.md`
mandates renaming. The choice between renaming the SGAI

  flagging the inconsistency as an erratum candidate.
- **§5.7 (lines 1230–1243)** — the operational §I.3.1 pair

## Open questions surfaced

- **The on-disk authoritative schema** for the URL-parameterisation
  child element (whether it is declared in the published
  `DASH-MPD.xsd` artefact under one name, the other, or both) is
  still not visible from the prose alone — Q2 of this audit
  confirmed the **printed** schema fragments omit both
  declarations. Resolving this requires consulting the published
  XSD file, which is outside the NotebookLM corpus available to
  this audit. The audit treats the prose-pair as the
  authoritative source and registers C-28 as Marginal on that
  basis.
- **Whether a single-name erratum has been filed against DASH 6th
  FDIS** for any of the schema-vs-prose splits (M-1, M-5) is
  outside the audit's scope. The spec's own footing — "issue
  inherited from DASH 6th; should be raised as an erratum
  candidate" — is the appropriate position for the SGAI
  specification.
- **The §5.16.5 "60s" default-value prose** for
  `@earliestResolutionTimeOffset` (Q1 finding) is a DASH 6th
  internal inconsistency: the cell reads *"The default value is
  60s (in units of timescale)"*, conflating seconds with
  timescale units. SGAI inherits this only on `<InsertPresentation>`
  / `<ReplacePresentation>` and explicitly sets the value in
  every worked example, so the inheritance does not propagate
  the confusion into the SGAI surface. Worth raising as a
  parallel erratum candidate against the MPEG-DASH editor.
- **N-1 remedy variants**: unchanged from v4.1 — three
  architectural options exist (Period-wrapper,
  AdaptationSet-wrapper, alternative envelope). This audit
  continues to recommend the Period-wrapper as the lowest-cost
  remedy.
