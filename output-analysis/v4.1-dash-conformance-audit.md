# DASH conformance audit — v4.1 (2026-05-15)

**Audit target**: `../output/v4.1-sgai-spec.md`.
**Reference**: MPEG-DASH 6th edition (ISO/IEC 23009-1:2025, FDIS),
  consulted via NotebookLM notebook "Streaming Protocols — DASH,
  HLS, C2PA, DRM" (the locally registered notebook bound to the
  `NOTEBOOK_NAME=streaming specs` setting in `.env.agent`).
**Method**: inventory of new / modified constructs introduced by
  the spec, NotebookLM queries grounded against the DASH 6th
  source, per-construct verdict
  (Conforming `V` / Marginal `M` / Non-conforming `N`).

## Scope

v4.1 is a **minor refinement** of v4. `context/` is unchanged;
the spec was regenerated via `prompts/refine-spec.prompt` to
apply remedies for items the v4 conformance audit, v4 detail
review, and v4 validation sidecar flagged as actionable inside
the minor envelope. The autofix surface in v4.1 (every refine
delta is annotated with an inline `<!-- refine: <issue-id> -->`
HTML comment) covers:

- **N-2** (v4 audit) — `<ReplacePresentation>` example XML changed
  from `clipDuration="<ticks>"` (schema-illegal: `@clipDuration`
  is `xs:boolean`) to a `returnOffset="0" startWithOffset="false"`
  pair that is schema-correct.
- **N-3** (v4 audit) — `<UrlParamInfo>` construct in Annex A.2,
  §2 normative references, and §5.7 was reshaped: the
  `<EssentialProperty schemeIdUri="urn:mpeg:dash:urlparam:2025"/>`
  is now empty and `<UrlParamInfo>` appears as a sibling in the
  core MPD namespace, consistent with the §I.3.1 prose.
- **M-1** (v4 audit) — schema-vs-prose `@url` / `@uri` split on
  `<InsertPresentation>` / `<ReplacePresentation>` now carries an
  inline NOTE in §5.1.1 and §5.1.2 documenting the inherited
  DASH-6th editorial split.
- **M-3** (v4 audit) — the `urn:svta:dash:sgai-layout:2026` row
  in §2.1 was dropped (it was declared but never used).
- **M-4** (v4 audit) — baseline DASH elements reused inside
  foreign-namespace subtrees (`<ImportedMPD>` and `<EventStream>`
  under `<svta:RenderableAsset>`) now carry inline NOTEs in §5.3.4
  and §5.5.3 marking the reinterpretation.

v4.1 also carries forward, by explicit declaration in the
"Refinement gaps" section, the v4 items the minor envelope could
not resolve — most notably **N-1** (Overlay Resolution Document
with zero `<Period>` elements) and **M-2**
(`@earliestResolutionTimeOffset` rebound unit on
`<svta:OverlayPresentation>` / `<svta:PauseAdPresentation>`). The
audit re-asserts the unchanged status of these items.

This audit is conformance-against-DASH-6th only. Design-level
audit lives in the v4.1 spec validation sidecar
(`v4.1-spec-validation.md`); detail review lives in
`v4.1-detail-review.md`.

- Inventory built from §2.1 (scheme URIs), §5.1 (presentation

## NotebookLM findings (highlights)

Three queries against the DASH 6th notebook ground the
substantive verdicts above.

**Q1 — `AlternativeMPDReplaceEventType` schema (verifies N-2 fix
in C-15).** NotebookLM quoted the schema fragment verbatim:

```xml
<xs:complexType name="AlternativeMPDReplaceEventType">
  <xs:complexContent>
    <xs:extension base="AlternativeMPDEventType">
      <xs:attribute name="returnOffset" type="xs:unsignedLong"/>
      <xs:attribute name="clipDuration" type="xs:boolean" default="true"/>
      <xs:attribute name="startWithOffset " type="xs:boolean" default="false"/>
    </xs:extension>
  </xs:complexContent>
</xs:complexType>
```

Key findings: `@returnOffset` is `xs:unsignedLong` (no default,
in `EventStream@timescale` units per Table 61); `@clipDuration` is
`xs:boolean` (default `true`) — confirming the v4 N-2 finding;
`@startWithOffset` is `xs:boolean` (default `false`). All three
attributes coexist on the same element (no `xs:choice`
restriction). The v4.1 autofix swapped the spec's
`clipDuration="<ticks>"` for `returnOffset="0"
startWithOffset="false"`, which the schema accepts. The trailing
space in the schema attribute name `"startWithOffset "` is a
DASH-6th typo unrelated to this audit.

**Q2 — §I.3.1 `<UrlParamInfo>` signalling rules (verifies N-3
fix in C-25).** NotebookLM quoted §I.3.1 verbatim: *"An
MPD.EssentialProperty element with the attribute @schemeIdUri
having value of 'urn:mpeg:dash:urlparam:2025' shall be present
and have no content. The scheme uses a single element,
UrlParamInfo of type ExtendedUrlInfoType (defined in Table I.3)
… UrlParamInfo element(s) may be present in elements such as
but not limited to MPD, Period, AdaptationSet, Preselection, or
EventStream."* The v4.1 autofix (empty `<EssentialProperty/>`
plus sibling `<UrlParamInfo>` in the core MPD namespace) matches
this prose exactly.

**Q3 — `UrlParamInfo` vs `RequestParam` (surfaces C-28).**
NotebookLM confirmed that *the same element has two names*
inside DASH 6th: Annex I.3.1 calls it `UrlParamInfo`; clause 5.3
semantic tables (Table 3, 4, 16, 44) call it `RequestParam`. The
two sections cross-reference each other (§I.3.1: "see clause 5.3
and Annex B"; clause 5.3 prose: "see Annex I.3 for details"). The
normative XML schema snippets in clause 5.3 omit the
`<xs:element>` declaration for this element, so the actual
on-disk schema name is not visible from the prose alone, but the
editorial centre of gravity lies on `RequestParam`. The v4.1
spec uses `<UrlParamInfo>` throughout — see C-28 for the marginal
finding.

## Non-conforming items detail

### N-1: Overlay Resolution Document has zero `<Period>` elements (C-20)

**Status: carried forward unchanged from v4 audit.** The v4.1
spec acknowledges the gap explicitly in the "Refinement gaps"
section (row 1) and defers the fix to the next major build.

The §5.2.2 boilerplate (lines 889–905), Annex C.3 (lines
1894–1971), and Annex E.3 (lines 2119–2149) all emit a `<MPD>`
whose only child is `<svta:OverlayList>` — zero `<Period>`
elements. Per the §5.3.1.2 / §5.3.1.3 schema, `<Period>` is
`maxOccurs="unbounded"` with implicit `minOccurs="1"`. A strict
DASH-conformant XML schema validator would reject the document.

**Why minor refinement could not lift this**: every proposed
remedy is an architectural change to the resolution-document
shape — wrap `<svta:OverlayList>` in a `<Period>`; place it
inside an AdaptationSet-bearing `<Period>`; or abandon the
`<MPD>` envelope altogether — and each rewrites §5.2.2, every
non-linear annex, and the `urn:svta:dash:profile:sgai-overlay-list:2026`
profile constraint. The Refinement-gaps row 1 framing is
correct: the choice between the three remedies is a context-
level architectural decision deferred to the next major build.

**Suggested fix for the next major build**: the cleanest of the
three is the "Period-wrapper" variant — declare a single empty
`<Period id="overlay-resolution" duration="P0S">` and place
`<svta:OverlayList>` as its child under §5.2.1 foreign-namespace
open content. This re-uses the established §5.2.1 extension path
already exercised everywhere else in the spec, makes the
document schema-conformant against `<Period minOccurs="1">`, and
costs only one syntactic level of nesting. (Variant trade-offs
are out of scope for this audit; see Refinement-gaps row 1.)

## Marginal items detail

### M-1: `@url` vs `@uri` on `<InsertPresentation>` / `<ReplacePresentation>` (C-14, C-15)

**Status: documentation surface improved in v4.1; underlying
issue carried forward.** v4.1 §5.1.1 (lines 641–652) and §5.1.2
(lines 668–672) now carry inline NOTEs spelling out the
DASH-6th editorial split: schema declares
`<xs:attribute name="uri" type="xs:anyURI" use="required"/>` on
`AlternativeMPDEventType`; Table 62 prose names the same
attribute `@url`. The NOTEs flag the inconsistency back to the
MPEG-DASH editor as an erratum candidate. A reader using a
strict schema validator is forewarned.

The spec continues to author `url="…"` in §5.1.1, §5.1.2, and
Annex A.2 (line 1705), so the schema validator would still
reject the example XML. The trade-off remains as in v4: prose-
fidelity to the linear SGAI baseline vs. schema fidelity. The
v4 verdict M is preserved with the documentation now in place.

### M-2: `@earliestResolutionTimeOffset` rebound unit on SVTA elements (C-13)

**Status: carried forward unchanged from v4 audit.** v4.1
preserves the §5.1.3 / §5.1.4 attribute-table NOTEs (lines 702,
786) flagging that the SVTA `@earliestResolutionTimeOffset` is
measured in `<EventStream>@timescale` units, NOT in seconds as
on `<ImportedMPD>`. The rebound unit aligns with §5.16.5 linear
SGAI baseline.

**Why minor refinement could not lift this**: the
context-level rule in `context/06-naming-and-namespaces.md`
mandates renaming when an attribute is reused with different
semantics. The choice between renaming the attribute everywhere
(e.g. `@earliestResolutionTimeOffsetTicks`) and amending
`context/06` to admit "inline NOTE" as an admissible audibility
lever is a context-level decision punted to the next major
build (Refinement-gaps row 2).

### M-3: `urn:svta:dash:sgai-layout:2026` declared but never used — **resolved in v4.1**

**Status: resolved by autofix in v4.1.** The v4 §2.1 table row
that introduced `urn:svta:dash:sgai-layout:2026` and the §3.2
anchor language that referenced it have both been dropped. The
v4.1 spec no longer declares the URI. M-3 is closed.

### M-4: Baseline DASH elements re-used inside foreign-namespace subtrees (C-17, C-19)

**Status: documentation surface improved in v4.1; underlying
verdict unchanged.** v4.1 adds two inline NOTEs documenting the
non-baseline placements:

- §5.3.4 (lines 1016–1024) on `<ImportedMPD>` as a child of
  `<svta:RenderableAsset>`: "this construct re-uses the baseline
  DASH `<ImportedMPD>` XML shape, but it is interpreted under
  SGAI semantics defined in this section, not under the baseline
  §5.3.2.6 placement rules". The NOTE explicitly addresses the
  question a baseline DASH editor would raise.
- §5.5.3 (lines 1104–1116) on the callback-scheme `<EventStream>`
  as a child of `<svta:RenderableAsset>`: "this construct re-uses
  the baseline DASH `<EventStream>` XML shape, but it is
  interpreted under SGAI semantics defined in this section …
  `<Event>@presentationTime` measured relative to the form-visible
  instant" instead of the baseline Period-start measurement.

Both NOTEs argue that the §5.2.1 fallback model strips the
parent `<svta:RenderableAsset>` together with its children, so
the baseline parser never reaches the re-used element — i.e.
the re-use is observable only by SGAI-aware Players. The
construct is schema-legal under the lax wildcard; the
reinterpretation is the design choice. The verdict remains
Marginal: a strict reading of the DASH schema still places
`<ImportedMPD>` under `<Period>` only, and the callback
`<EventStream>` likewise; v4.1's NOTEs document this rather
than restructure the spec to avoid it.

The v4.1 validation sidecar item A-5 ("context-level
authorisation for the baseline-DASH-element-under-SVTA-parent
pattern") is the residual debt: a context rule in
`context/08-dash-extension-rules.md` would close it; that is a
major-build addition (Refinement-gaps row 15).

### M-5 (new): `<UrlParamInfo>` vs `<RequestParam>` element name (C-28)

**Status: surfaced by the v4.1 N-3 autofix.** The v4 audit did
not query NotebookLM with this granularity; the v4.1 audit does
(Q3). DASH 6th carries two names for the same element across
its own sections:

- **Annex I.3.1** names it `UrlParamInfo` of type
  `ExtendedUrlInfoType` (Table I.3).
- **Clause 5.3 core semantic tables** name it `RequestParam` —
  Table 3 (under `<MPD>`, 0..N), Table 4 (under `<Period>`,
  0..1), Table 16 (under `RepresentationBaseType`, 0..N),
  Table 44 (under `<EventStream>`, 0..N).

The two sections cross-reference each other. The normative XML
schema snippets in clause 5.3 omit the `<xs:element>`
declaration, so the on-disk schema name cannot be confirmed
from the prose alone, but the structural editorial gravity sits
on `RequestParam` (it is the name in every semantic table that
positions the element in the hierarchy).

The v4.1 spec follows §I.3.1 prose (`<UrlParamInfo>`) throughout
— §2 normative references (line 114), §5.7 (lines 1204–1216),
Annex A.2 (line 1698). A strict-schema-validator tool that
treats the clause 5.3 tables as authoritative may reject the
`<UrlParamInfo>` element name; a tool that uses the §I.3.1
prose name will accept it.

This is the same schema-vs-prose pattern as M-1
(`@url` / `@uri`) and the same mitigation is recommended:

- **Suggested fix**: add an inline NOTE in §5.7 and in the
  §2 normative references row analogous to the M-1 NOTEs in
  §5.1.1 / §5.1.2 — *"NOTE: DASH 6th carries this element
  under two names; the clause 5.3 semantic tables (Table 3, 4,
  16, 44) name it `RequestParam` while §I.3.1 prose names it
  `UrlParamInfo`. This specification follows the §I.3.1 prose
  name. The inconsistency is inherited from DASH 6th and
  should be raised against the MPEG-DASH editor as an erratum
  candidate."* No structural change to the example XML is
  required; the NOTE alone closes the M-5 documentation gap.
  The fix is small enough to fit a future minor refinement
  (v4.2).

## Open questions surfaced

- **The on-disk authoritative schema name** for the
  `<UrlParamInfo>` / `<RequestParam>` element is not visible from
  the prose in clause 5.3 because the `<xs:element>` declaration
  is omitted from the quoted schema snippets (Q3). Resolving
  this requires consulting the published `DASH-MPD.xsd`
  artefact, which is outside the NotebookLM corpus available to
  this audit. The audit treats the prose-pair as the
  authoritative source and registers C-28 as Marginal rather
  than Non-conforming on that basis.
- **Whether a single-name erratum has been filed against DASH 6th
  FDIS** for any of the schema-vs-prose splits (M-1, M-5) is
  outside the audit's scope. The spec's own footing — "issue
  inherited from DASH 6th; should be raised as an erratum
  candidate" — is the appropriate position for the SGAI
  specification.
- **N-1 remedy variants**: three architectural options exist
  (Period-wrapper, AdaptationSet-wrapper, alternative envelope).
  This audit recommends the Period-wrapper as the lowest-cost
  remedy in the suggested fix above, but the final choice is a
  context-level decision for the next major build.
